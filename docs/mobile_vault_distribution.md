# 移动端知识库分发方案（鸿蒙 ArkTS）

目标：移动端不包含联网 AI，仍做到"安装即能用、可导入自己的完整库、可与桌面端同步"。

## 关键概念：多知识库（vaultId）

移动端和桌面端均支持**多个知识库**。每个知识库在服务端和本地都有唯一的 `vaultId`：

- 桌面端：在 WebUI「配置」→「知识库设置」中添加，系统自动分配 `vaultId`
- 移动端：导入/同步时按 `vaultId` 隔离存储，数据库和文件系统均以 `vaultId` 作为命名空间

同步时，移动端必须显式选择当前目标知识库（`currentVaultId`），所有 API 调用都携带该 `vaultId`。

## 两种方案

### 方案 A：空库 + 引导导入（主方案）

- 安装后移动端只有一个空的 vault 目录（应用内沙箱）
- 首次启动展示引导：
  - 选"导入我的知识库"（系统文件选择器）
  - 选"使用示例库（小样本）"（从 assets 解压到应用目录）
  - 选"稍后再说"（进入空库，只能看到引导页）

**优点**

- 不分发大规模内容，降低版权/内容合规风险
- 包体积小、迭代快

**风险/注意**

- 必须提供"系统选择器导入"与明确引导，否则审核容易认为不可用/体验差

### 方案 B：内置完整知识库（后置方案）

- 将完整 vault 作为 assets 随包发布，首次启动解压到应用目录

**优点**

- 首次即有内容，体验最佳

**主要风险**

- 版权/授权材料：内容若非完全原创，审核可能要求权属证明
- 包体积：过大可能影响下载、安装、审核体验
- 健康/医疗内容合规：需要更强的免责声明与功能边界说明

> 建议：先用"小示例库"跑通上架与体验，再评估是否内置完整库。

## "小示例库"建议规格（用于上架版）

- 规模：**30–80 条** Markdown
- 覆盖：症状/证型/病机/方剂等学习笔记类条目（各 5–20 条）
- 保留：少量 `[[link]]` 展示双向链接体验
- 目的：证明应用"可用"、展示检索与条目详情

示例库源可以复用现有 Flutter 工程里的 `assets/seed_vault/`（已存在）。

## 导入/导出格式（建议统一 ZIP）

### 导入

- 用户从系统选择器选择一个 `.zip`
- 应用将 zip 解压到应用内对应 `vaultId` 的目录（可选：解压前清空/合并策略）
- 导入完成后，该 `vaultId` 即作为本地独立知识库可用

### 导出

- 将当前选定 `vaultId` 对应的应用内目录打包为 zip
- 用户选择保存位置（系统选择器）

### ZIP 规范（建议）

- 根目录必须包含 `manifest.json`（便于版本/校验/合并策略）
- 内容以相对路径存放，例如：
  - `files/症状/头痛.md`
  - `files/证型/桂枝汤证.md`

`manifest.json` 示例：

```json
{
  "formatVersion": 1,
  "app": "doctor-notes",
  "vaultId": "my-vault-001",
  "createdAt": "2026-03-18T00:00:00Z",
  "fileCount": 123,
  "note": "exported from mobile"
}
```

> 第一版可以不做 hash/签名，后续与"同步协议"打通时再加。

## 与桌面端同步的关系

- 同步协议见 `docs/sync_protocol.md`
- 同步前，移动端必须在「知识库管理」中选中一个 `vaultId` 作为当前目标
- 桌面端和移动端的 `vaultId` 必须一致，才能正确对应到同一个知识库
- 移动端可先实现"导入/导出 ZIP"，与桌面端同一 `vaultId` 的增量同步作为第二阶段
